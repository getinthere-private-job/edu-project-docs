```
┌────────────────────────────────────────────────────────────┐
│                                                            │
│  분류 1. PBL 프로젝트 문서                                   │

│                                                            │

│  - 이론평가: 객관식 시험 → 채점지 PDF                          │

│  - 자동생성: 성적분석표, 성적통계, 총평 등 보고서 PDF             │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

## 단일 진입점: 홈페이지 대시보드

```
작성은 각자 편한 곳에서, 확인은 한 곳에서!

┌─────────────────────────────────────────────────────────┐
│                홈페이지 대시보드 (단일 진입점)                │
│                                                         │
├─────────────────┬─────────────────┬─────────────────────┤
│ PBL 프로젝트 문서 │ 시험 문서        │ 소스코드             │
│ (출력 불필요)     │ (PDF 출력 필요)  │                     │
│                 │                 │                     │
│ 강사 입력 → 보기  │ 시험 → PDF 생성  │ GitHub 링크          │
│ 노션 → 링크 이동  │ [PDF 다운로드]   │ [GitHub 열기]        │
└─────────────────┴─────────────────┴─────────────────────┘
       ↑                  ↑                  ↑
  강사: 홈페이지 입력   홈페이지에서 시험     학생: GitHub
  학생: 노션 작성
```

### 작동 방식

1. **팀 등록 시** 노션 URL + GitHub URL을 홈페이지에 입력
2. **대시보드**에서 PBL 문서 + 시험 문서를 한 화면에 표시
3. PBL 프로젝트 문서 → 홈페이지 내부 보기 or 노션 링크 이동 (출력 불필요)
4. 시험 문서 → PDF 다운로드 (출력 필요)
5. 소스코드 → GitHub 링크 이동

## 도구별 역할

```
홈페이지 (NCS평가사이트)           노션                GitHub
━━━━━━━━━━━━━━━━━━━━━━━━━━━    ━━━━━━━━━━━━━━━━    ━━━━━━━━━━━━━━━━
1. 시험 (PDF 필요)               학생이 작성           학생이 코드 관리
   - 객관식 시험 → 채점지 PDF
   - 포트폴리오 평가 → 채점지 PDF
   - 보고서 5종 자동생성 PDF

2. PBL 프로젝트 강사 입력 (출력 불필요)
   - 관찰일지 입력 → 홈페이지에서 보기
   - 평가표 입력 → 홈페이지에서 보기
   - 멘토링일지 입력 → 홈페이지에서 보기

3. 대시보드 (단일 진입점)
   - PBL 문서 현황 + 노션/GitHub 링크
   - 시험 PDF 다운로드
━━━━━━━━━━━━━━━━━━━━━━━━━━━    ━━━━━━━━━━━━━━━━    ━━━━━━━━━━━━━━━━
```

### 핵심 원칙

- **PBL 프로젝트 문서** = 출력 불필요, 노션/홈페이지에 존재만 하면 됨
- **시험 문서** = PDF 출력 필요, 홈페이지에서 시험 → 자동생성
- **확인** = 홈페이지 대시보드 하나로 전부 확인 (링크 + PDF 다운로드)

## 사용 시나리오

### 강사 시나리오

```
프로젝트 시작 전
│
├─ [홈페이지] 과정/교과목/학생/팀 등록
│   └─ 팀 등록 시 노션 URL, GitHub URL 입력
├─ [홈페이지] 시험지 등록 (객관식/포트폴리오)
│
프로젝트 진행 중 (4주)
│
├─ 매일
│   ├─ 스탠드업 참관
│   ├─ [대시보드 → 노션 링크] 학생 회의록, 하브루타 근거자료 확인
│   └─ [대시보드 → GitHub 링크] 커밋 현황 확인
│
├─ 스텝 완료시
│   └─ [GitHub] master 커밋 코드리뷰 (노션에 리뷰 기록은 학생이)
│
├─ 주 1회
│   └─ [홈페이지] 관찰일지 입력 (출력 불필요, 홈페이지에서 보기만)
│       (학생별 대인관계/문제해결 점수 + 근거자료 체크)
│
├─ 멘토링 (4회)
│   ├─ 멘토의 GitHub 소스코드 리뷰 진행 (2~3회차)
│   └─ [홈페이지] 멘토링일지 입력 (출력 불필요, 홈페이지에서 보기만)
│       (질문/답변, 피드백, 코드리뷰 결과, 액션아이템)
│
├─ 중간 발표
│   ├─ [홈페이지] 프로젝트 평가표 입력 (출력 불필요, 홈페이지에서 보기만)
│   ├─ [홈페이지] 객관식 시험 진행 → 채점지 PDF 자동생성
│   └─ [홈페이지] 포트폴리오 평가 진행 → 채점지 PDF 자동생성
│
└─ 최종 발표
    ├─ [홈페이지] 프로젝트 평가표 입력
    ├─ [홈페이지] 객관식/포트폴리오 시험 → 채점지 PDF 자동생성
    └─ [대시보드] 전체 확인
        ├─ PBL 문서 → 링크로 확인 (출력 불필요)
        └─ 시험 PDF → 일괄 다운로드 (출력 필요)
```

### 학생 시나리오

```
프로젝트 시작
│
├─ 1주차
│   ├─ [노션] PBL 프로젝트 계획서 작성
│   └─ [GitHub] 레포 생성, 브랜치 전략 설정
│
├─ 매일
│   ├─ [노션] 회의록 작성 + 회의 사진 첨부
│   ├─ [노션] 하브루타노트 작성 (개인)
│   └─ [GitHub] feature 브랜치에서 코드 작업
│
├─ 스텝 완료시 (회원가입, CRUD, 핵심기능 등)
│   ├─ [GitHub] master에 커밋 → 강사 코드리뷰 받음
│   └─ [노션] 코드리뷰 결과 기록
│
├─ 에러 발생시
│   └─ [노션] 트러블슈팅 기록 (증상→원인→해결→배운점)
│
├─ 수시
│   └─ [노션] 블로그 작성 → 노션 복제, 주 1회 발표
│
├─ 스프린트마다
│   └─ [노션] PBL 프로젝트 계획서 업데이트 + KPT 회고
│
├─ 중간/최종
│   └─ [홈페이지] 시험 응시 (객관식/포트폴리오)
│
└─ 종료
    ├─ [노션] 프로젝트회고 작성 + 모든 문서 정리
    └─ [노션] 최종산출물 준비 (PPT, 시연영상, 발표영상)
```

## 코드리뷰 방식

### 실제 운영 방식
```
학생: feature 브랜치에서 자유롭게 작업
      ↓
      특정 스텝 완료 (예: 회원가입, 핵심 CRUD, Redis 등)
      ↓
      master에 커밋
      ↓
강사: master 커밋만 코드리뷰
      ↓
학생: 리뷰 결과를 노션 코드리뷰 문서에 기록
```

### 코드리뷰 대상 스텝 (예시)

| 스텝 | 시기 | 리뷰 포인트 |
|------|------|-----------|
| 프로젝트 초기 설정 | 1주차 | 패키지 구조, 설정 파일 |
| 회원 관리 | 1~2주차 | 엔티티 설계, 인증 로직 |
| 핵심 도메인 CRUD | 2주차 | JPA 연관관계, API 설계 |
| Redis 세션 | 2~3주차 | 세션 관리 구현 |
| 추가 기능 | 3~4주차 | 코드 품질, 예외처리 |
| 최종 정리 | 4주차 | 전체 코드 구조 |

## 멘토 워크플로우

> 각 프로젝트에는 참여기업 멘토가 배정됩니다.
> 멘토는 프로젝트 전 과정에 걸쳐 조언, 코드리뷰, 평가에 참여합니다.

### 멘토 역할

| 시점 | 활동 | 상세 |
|------|------|------|
| **프로젝트 시작** | 프로젝트 조언 | 주제 적합성, 아키텍처 설계, 요구사항에 대한 실무 관점 피드백 |
| **진행 중 (2~3회)** | GitHub 소스코드 리뷰 | GitHub 저장소의 코드를 직접 리뷰하고 개선점 피드백 |
| **프로젝트 종료** | 최종 리뷰 | 최종 결과물 평가, 실무 피드백, 커리어 조언 |

### 멘토 시나리오

```
프로젝트 시작 (1차 멘토링)

│   ├─ 기술 이슈 해결 지원
│   └─ 리뷰 결과 → 멘토링일지 + 학생 코드리뷰 문서에 기록
│
├─ 중간 발표 참관 및 피드백
│
프로젝트 종료 (4차 멘토링)
│
├─ 최종 결과물 리뷰 및 평가
├─ 실무 관점 종합 피드백
└─ 커리어 조언
```
