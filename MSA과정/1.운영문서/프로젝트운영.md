| 항목 | 내용 |

| 훈련시간 | 140시간 (약 4주) |

### 1-2. PBL 학습 목표

| 구분 | 학습 목표 |
|-----|----------|
| 기술 역량 | Spring Boot 기반 모놀로식 웹 애플리케이션 설계 및 구현 |

| 실무 역량 | 요구사항 분석 → 설계 → 구현 → 테스트의 개발 사이클 경험 |

### 1-3. PBL 운영 원칙

> PBL(Problem/Project Based Learning)은 학생이 실제 문제를 중심으로 자기주도적으로 학습하는 방식입니다.
> 강사는 **퍼실리테이터(Facilitator)** 역할을 하며, 학생이 스스로 문제를 발견하고 해결하도록 안내합니다.

| 원칙 | 설명 |
|-----|------|
| 자기주도 학습 | 배우지 않은 기술도 직접 조사·학습하여 프로젝트에 적용 (예: Redis 세션, SSE, API 설계 등) |
| 문제 중심 | 실제 서비스를 만든다는 가정 하에 요구사항을 도출하고 문제를 해결 |
| 협력 학습 | 팀원 간 역할 분담, 코드리뷰, 하브루타 토론을 통한 상호 학습 |
| 반복적 피드백 | 주간 스프린트 단위로 멘토링 → 피드백 → 개선 사이클 반복 |
| 결과물 발표 | 중간/최종 발표를 통해 학습 내용을 정리하고 공유 |

| **4단계: 중간 점검 및 피드백** | 11일 (8h) | - 중간 발표 (팀별 10~15분)<br>- 멘토 피드백<br>- 개선 계획 수립 | - 중간 평가<br>- 멘토링<br>- 관찰일지 기록 | - 중간 발표 자료<br>- 멘토링 피드백 기록<br>- 개선 계획서 |

### 1-5. PBL에서 학생이 자기주도 학습해야 하는 영역

> PBL 방식이므로 아래 기술은 수업에서 직접 가르치지 않으며, 학생이 프로젝트 진행 중 필요에 의해 스스로 학습합니다.

| 영역 | 자기주도 학습 항목 | 학습 참고 자료 |
|-----|-----------------|--------------|
| 세션 관리 | Redis를 활용한 세션 클러스터링 | Spring Session Redis 공식 문서 |
| 템플릿 엔진 | Mustache 문법 및 레이아웃 구성 | Mustache 공식 GitHub, Baeldung |
| 실시간 통신 | SSE(Server-Sent Events)를 활용한 알림 서비스 | Spring SseEmitter 공식 문서 |
| API 설계 | RESTful API 설계 원칙 및 적용 | REST API Tutorial, 기술 블로그 |
| 테스트 | JUnit5를 활용한 JPA Repository 쿼리 테스트 | JUnit5, @DataJpaTest 공식 문서 |
| 버전 관리 | Git Flow 전략, PR 기반 코드리뷰 | Atlassian Git Flow 가이드 |
| 보안 (선택) | Spring Security 기초, CSRF, XSS 방지 | Spring Security 공식 문서 |

### 1-6. 팀 구성 기준

## 2. PBL 프로젝트 계획서

> PBL 프로젝트 계획서는 **강사가 최소 요구사항 템플릿을 제공**하고, **학생이 자신의 주제에 맞게 작성·제출**합니다.
> 학생은 프로젝트 초기에 작성하고, 스프린트마다 업데이트합니다.

> 강사는 아래 최소 요구사항 템플릿만 제공합니다.
> 학생은 이 템플릿을 기반으로 자신의 프로젝트에 맞는 PBL 프로젝트 계획서를 작성합니다.

#### 1. 프로젝트 개요

부산 지역 맛집 정보를 통합하여 사용자 취향에 맞는 맛집을 추천하는 웹 서비스.
사장님은 가게를 등록하고, 고객은 리뷰를 작성하며, 카테고리/지역별 검색이 가능합니다.

#### 2. 문제 정의

- 맛집 정보가 블로그, SNS 등에 분산되어 있어 신뢰도 높은 정보를 찾기 어려움
- 지역 기반 맛집 검색 + 리뷰를 한 곳에서 확인할 수 있는 플랫폼 필요
- 사장님이 직접 가게 정보를 관리할 수 있는 기능 필요

#### 3. 기능 목록

**MVP (필수)**

| ID | 기능 | 설명 | 우선순위 |
|----|------|------|---------|
| F-01 | 회원가입/로그인 | 일반회원, 사장님, 관리자 3개 역할 | 필수 |
| F-02 | 맛집 등록/수정/삭제 | 사장님이 자기 가게 CRUD | 필수 |
| F-03 | 맛집 목록/상세 조회 | 카테고리, 지역별 필터 + 페이징 | 필수 |
| F-04 | 리뷰 작성/조회 | 별점 + 텍스트 리뷰 | 필수 |
| F-05 | 검색 | 가게명, 카테고리, 지역 검색 | 필수 |

**2단계 (선택)**

| ID | 기능 | 설명 | 우선순위 |
|----|------|------|---------|
| F-06 | 찜하기 | 관심 맛집 찜 목록 | 중요 |
| F-07 | 실시간 알림 | 새 리뷰 알림 (SSE) | 선택 |
| F-08 | 관리자 대시보드 | 통계, 가게 승인 관리 | 선택 |

#### 4. 기술 스택 선정 이유

| 기술 | 선정 이유 |
|-----|----------|
| Spring Boot 3.x | 수업에서 학습한 프레임워크, 생산성 높은 웹 개발 |
| MySQL 8 | 관계형 데이터 (가게-리뷰-회원) 구조에 적합 |
| JPA/Hibernate | 객체-관계 매핑으로 개발 생산성 향상 |

| Redis | 세션 클러스터링, 서버 재시작 시에도 세션 유지 |
| Git/GitHub | 팀 협업, 코드리뷰, 이력 관리 |

#### 5. DB 설계 초안 (ERD)

```
Member (회원)
├── member_id (PK)
├── email, password, nickname
├── role (CUSTOMER / OWNER / ADMIN)
└── created_at, updated_at

Restaurant (맛집)           ← Owner가 등록
├── restaurant_id (PK)
├── owner_id (FK → Member)
├── name, description, address, phone
├── category_id (FK → Category)
└── created_at, updated_at

Category (카테고리)
├── category_id (PK)
└── name (한식, 중식, 일식, 양식, 카페 등)

Review (리뷰)               ← Customer가 작성
├── review_id (PK)
├── member_id (FK → Member)
├── restaurant_id (FK → Restaurant)
├── rating (1~5), content
└── created_at

Bookmark (찜)               ← 2단계
├── bookmark_id (PK)
├── member_id (FK → Member)
└── restaurant_id (FK → Restaurant)
```

**연관관계**: Member 1:N Restaurant (사장님), Member 1:N Review, Restaurant 1:N Review, Category 1:N Restaurant

### 2-4. 팀 운영 규칙 (필수)

> PBL 프로젝트에서 팀 협업의 질을 높이기 위한 필수 운영 규칙입니다.
> 아래 항목은 모든 팀이 반드시 지켜야 하며, 국가 제출 근거자료로 활용됩니다.

| 번호 | 운영 규칙 | 상세 내용 | 근거자료 |
|-----|----------|----------|---------|
| 1 | **정기 회의 시간 운영** | 팀별로 회의 시간을 정해놓고 매일 진행 (예: 매일 09:30~09:40 스탠드업) | 회의록 (노션) |
| 2 | **회의록 작성** | 모든 회의는 기록으로 남기기. 노션에 회의록 템플릿으로 작성 | 회의록 (노션) |
| 3 | **회의 사진 촬영** | 매 회의 시 팀원이 함께 있는 사진 1장 남기기 (근거자료) | 사진 (노션에 첨부) |
| 4 | **팀 블로그 운영** | 모르는 기능을 학습하면 블로그에 작성하고, 노션에 복제하여 모아두기 | 팀 블로그 + 노션 링크 |
| 5 | **팀 블로그 발표** | 주 1회 팀별로 블로그 작성 내용을 발표하는 시간 가지기 | 발표 기록 (노션) |
| 6 | **MVP 정의 및 DoD** | 반드시 MVP를 정하고, DoD(Definition of Done) 기준을 명확히 하기 | PBL 프로젝트 계획서 (노션) |
| 7 | **트러블슈팅 기록** | 에러/문제 발생 시 증상→원인→해결→배운점을 노션에 기록. GitHub Permalink로 코드 링크 첨부 | 트러블슈팅 문서 (노션) |
| 8 | **코드리뷰** | 특정 스텝 완료 후 master 커밋 → 강사 리뷰 → 노션에 리뷰 결과 기록 | 코드리뷰 기록 (노션) |

### 2-5. 요구사항 우선순위 분류

| 우선순위 | 설명 | 예시 |
|---------|------|------|
| **필수 (Must)** | 프로젝트 핵심 기능, 없으면 서비스 불가 | 회원관리, 핵심 CRUD, Redis 세션, MVP 범위 기능 |
| **중요 (Should)** | 서비스 품질에 큰 영향, 가능한 포함 | 페이징, 예외처리, 유효성 검증 |
| **선택 (Could)** | 있으면 좋지만 없어도 기본 동작 가능 | 파일 업로드, SSE 알림, Spring Security, 검색 |
| **추후 (Won't)** | 이번 프로젝트에서는 제외, 2차 MSA에서 고려 | MSA 전환, 컨테이너화, CI/CD |

### 4-7. 평가 체계 연계

> 프로젝트에는 **2종의 평가**가 존재합니다. 혼동하지 않도록 아래 구분을 확인하세요.

| 구분 | 프로젝트 평가표 (본 섹션) | 포트폴리오 평가 (시험 문서) |
|------|------------------------|------------------------|
| **성격** | PBL 프로젝트 내부 평가 | HRD 공식 실습평가 |
| **구조** | 팀 50점 + 개인 50점 | 6영역 100점 (계획/내용/수준/기능수준/태도/구두발표) |
| **작성** | 강사가 홈페이지에 입력 | 강사가 홈페이지에서 채점 |
| **출력** | 불필요 (대시보드에서 보기) | PDF 출력 필요 |
| **시기** | 중간/최종 발표 후 | 중간/최종 시험 시 |
| **근거자료** | 노션 + GitHub 링크 | 프로젝트 평가표 + 발표 현장 |

**연계 방식**: 프로젝트 평가표의 팀/개인 점수를 근거로 포트폴리오 평가(HRD 6영역)를 채점합니다.

| HRD 6영역 | 배점 | 프로젝트 평가표에서 참고 |
|-----------|------|----------------------|
| 계획 | 20점 | 팀: 계획 (10점) |
| 내용 | 20점 | 팀: 구현 - MVP 완성도, 기술스택 활용도 (14점) |
| 수준 | 15점 | 팀: 구현 - 코드 품질 (6점) + 개인: 지식 - 기능 구현 수준 (8점) |
| 기능수준 | 15점 | 팀: 산출물 (10점) |
| 태도 | 10점 | 개인: 태도 (20점) + 성장 (10점) 종합 |
| 구두발표 | 20점 | 팀: 발표 (10점) |

> **관찰일지와의 관계**: 관찰일지(주차별 대인관계/문제해결 점수)는 프로젝트 평가표 작성 시 근거자료로 활용됩니다.
> 관찰일지의 누적 기록 → 개인 평가(태도/지식/성장) 채점 시 참고.

## 부록: 단일 진입점 - 홈페이지 대시보드

> **작성은 각자 편한 곳에서, 확인은 홈페이지 한 곳에서!**

### 작동 방식

1. **팀 등록 시** 노션 URL + GitHub URL을 홈페이지에 입력
2. **대시보드** 한 화면에서 PBL 문서 + 시험 문서 전부 표시
3. PBL 프로젝트 문서 → 홈페이지 보기 or 노션/GitHub 링크 이동 (출력 불필요)
4. 시험 문서 → PDF 다운로드 (출력 필요)

### 도구별 역할

| 작성 주체 | 관리 도구 | 관리 문서 | 대시보드 연결 |
|----------|----------|----------|-------------|
| 강사 | **홈페이지** | 시험(PDF) + PBL 강사 입력(보기만) | PDF 다운로드 / 보기 |
| 학생 | **노션** | PBL 학생 문서 8종 | 노션 링크로 연결 |
| 학생 | **GitHub** | 소스코드, 스텝 완료 시 master 커밋 | GitHub 링크로 연결 |

> 학생 문서 작성 가이드: `3.학생문서/노션가이드.md` 참조
> 전체 워크플로우: `workflow.md` 참조
